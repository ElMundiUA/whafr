## Architectural finding

The security+FE re-probe agent spotted the load-bearing issue:

> Phases 1-4 (OWASP, oauth.net RFCs, web.dev/vitals, react.dev,
> shadcn/ui, MDN, Chrome dev, W3C) were *ingested* but are **not
> retrievable** for the canonical queries security and frontend
> engineers actually ask. The retrieval layer returns
> **entity-graph relationship snippets** — "X requires Y",
> "A authored B" — rather than **document body chunks**. WCAG/ARIA
> and Carbon work because their entities *are* the answer; everything
> that needs procedural / prescriptive prose (CSP directives, PKCE
> flow steps, LCP thresholds, hook semantics) returns adjacency
> trivia.

This is a Graphiti-design constraint: `client._search()` searches and
returns `EntityEdge` rows. The raw Episodic content (the article body)
is stored but never returned via search — only via the explicit
`fetch_source(episode_id)` tool, which requires the agent to know
which episode to look at first.

Two ways to fix:

1. **Add episode-body retrieval** as a second pass behind `search`.
   Either via the `COMBINED_HYBRID_SEARCH_CROSS_ENCODER` recipe
   (which includes episode BM25), or a direct Cypher fulltext query
   on Episodic.content. The result list would mix one-line facts with
   100-300 word excerpts from the source document.
2. **Tune Graphiti's fact-extraction** to emit fine-grained,
   procedural facts (e.g. for an OWASP page on SQLi, emit edges like
   "Parameterized Queries → prevent → SQL Injection via bind
   variables"). Today extraction stops at entity-level relationships.

Path 1 is the bigger lever and less coupling-y. **Filed as a
follow-up before measuring real gap drop on procedural domains.**

text**, not entity-fact triples. This is the architecture finding
above.
